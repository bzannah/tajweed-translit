MANDATORY EXECUTION SEQUENCE

PHASE 1 — BASELINE, INVENTORY, AND BLOCKERS

A. Build a full SEO inventory
- Enumerate all routes, templates, content types, taxonomies, filter/facet pages, parameterized URLs, archives, media assets, docs sections, search pages, and utility pages.
- Determine which URLs should be indexable, canonical, merged, redirected, improved, or noindexed.
- Detect duplicate content, near-duplicates, cannibalization, thin pages, orphan pages, soft 404s, redirect chains, broken canonicals, broken internal links, and low-value archives/tags.
- Compare the current URL set to the desired canonical URL set.

B. Establish a measurement baseline
- Collect or prepare baselines for indexed pages, impressions, clicks, CTR, average position, top queries, top landing pages, conversions from organic, crawl errors, page speed / Core Web Vitals, schema coverage, internal-link depth, and template performance.
- If Search Console / GA4 / logs are missing, set them up or provide exact setup steps and create interim measurements from the site and repo.

C. Find the highest-leverage blockers
- Technical: crawl traps, blocked resources, JS rendering failures, hydration-only content, canonical errors, hreflang errors, poor sitemap coverage, status-code problems, weak metadata systems, schema errors, image delivery problems, broken mobile UX, security/spam page issues, slow templates, CLS/INP/LCP issues.
- Strategic: wrong keyword targeting, intent mismatch, missing money pages, shallow cluster coverage, undifferentiated copy, weak trust signals, poor conversion paths, lack of first-party evidence, or fragmented information architecture.

PHASE 2 — QUERY STRATEGY AND INFORMATION ARCHITECTURE

1. Query universe
- Build the full query universe around the product/topic.
- Cluster by intent: navigational, informational, commercial investigation, transactional, support/docs, comparison, alternatives, use-cases, jobs-to-be-done, and problem/solution queries.
- Identify the single best canonical page to own each cluster.

2. Site architecture
- Design the cleanest possible hub-and-spoke structure for priority clusters.
- Define hubs, spokes, comparison pages, alternative pages, solution pages, use-case pages, docs/tutorials, templates/tools/calculators, glossary pages, and category/collection pages as relevant.
- Design URL rules, breadcrumb rules, canonical rules, and internal-link routes that make the site easy to crawl and easy to understand.

3. Competitive moat
- Define why this site deserves to outrank existing results.
- Prefer first-party data, real screenshots, product evidence, expert explanations, walkthroughs, benchmarks, examples, templates, tools, case studies, integration depth, comparison tables, FAQs, and task-completion design.
- Mine repo docs, changelogs, issues, release notes, support language, and existing user vocabulary for real query language and pain points.

PHASE 3 — IMPLEMENTATION

A. Crawlability, indexation, canonicalization
- Fix or create robots.txt, XML sitemaps, canonical tags, redirects, hreflang where applicable, pagination handling, parameter handling, meta robots / X-Robots-Tag behavior, and correct status codes.
- Make sure every important page is linked internally with crawlable HTML links.
- Remove or consolidate thin, duplicate, placeholder, or low-value pages.
- Ensure critical content is server-rendered / statically rendered / prerendered when needed for reliable discovery and rendering.

B. Internal linking and hierarchy
- Create a deliberate internal-linking system using hubs, breadcrumbs, contextual links, related pages, “best next step” modules, comparison tables, and sibling-page discovery.
- Reduce click depth to priority pages.
- Use natural, descriptive anchors that reflect user language.
- Assign one dominant page to each query cluster and resolve cannibalization.

C. On-page execution
For every priority page:
- Define the page’s primary query cluster and intent.
- Write or improve the title element, H1, supporting headings, opening summary, body structure, comparison/decision aids, FAQ sections where genuinely useful, and the CTA/next step.
- Make the content better, not just longer.
- Add definitions, examples, alternatives, objections, edge cases, and decision support that help the user finish the job.
- Use the exact language real users search for naturally in titles, headings, alt text, and link text.
- Engineer snippet-friendly sections: concise summaries, definition paragraphs, ordered steps, comparison tables, short answer blocks, and scannable structure where it helps users.
- Ensure meta descriptions are distinct, compelling, and aligned to the visible content.
- Add honest freshness signals when relevant, such as changelogs or updated dates tied to real updates.

D. Structured data
- Add only truthful, page-relevant JSON-LD that matches visible content.
- Use the best-fitting supported types for the page templates, such as Organization, BreadcrumbList, Product, SoftwareApplication, Article, VideoObject, LocalBusiness, Event, Dataset, DiscussionForumPosting, JobPosting, Course, FAQPage, HowTo, Review, or AggregateRating only when genuinely eligible.
- Generate schema from source data where possible so it stays synchronized.
- Validate and fix all errors/warnings that materially affect eligibility or trust.
- Never mark up invisible, misleading, irrelevant, or fabricated content.

E. Images, media, and asset SEO
- Improve image filenames, alt text, captions, surrounding text, dimensions, compression, lazy-loading, and responsive delivery.
- Make media pages and embedded assets discoverable only when they deserve indexation.
- Add video markup/transcripts where relevant and useful.

F. Performance and UX
- Improve Core Web Vitals and mobile usability on key templates.
- Optimize rendering strategy, HTML response size, critical CSS, JS execution, font loading, image delivery, caching, and interaction latency.
- Remove layout shifts and high-friction UI patterns on landing pages.
- Preserve semantic HTML and accessibility.

G. Content system and scale
- Build reusable SEO-safe page templates, metadata systems, schema components, and editorial briefs.
- Create a prioritized backlog of pages to improve, consolidate, or create.
- If programmatic SEO is appropriate, only generate pages when each page has unique data, unique utility, unique intent, and a clear reason to exist.
- Never mass-produce near-duplicate pages just to target keywords.

H. Trust, proof, and task completion
- Add real proof wherever available: product screenshots, integrations, benchmarks, pricing clarity, docs depth, support/contact clarity, security/privacy info, testimonials/case studies, expert authorship, and company/about information.
- Make sure the page actually solves the user’s problem after the click.

I. Authority and link earning
- Create or plan link-worthy assets: tools, studies, benchmarks, open data, templates, explainers, calculators, integration pages, launch pages, or community resources.
- Draft ethical outreach or distribution ideas when appropriate.
- Do not buy links or pursue manipulative link schemes.

J. QA automation and SEO guardrails
- Add or improve automated checks for broken links, missing titles/meta, duplicate canonicals, noindex mistakes, orphan pages, sitemap integrity, schema validity, and performance budgets where possible.
- Add CI/CD or pre-publish checks for high-risk SEO regressions.

PHASE 4 — MEASUREMENT, DASHBOARDS, AND EXPERIMENTS
- Set up or improve Search Console, GA4, server-log analysis, schema validation, and performance monitoring.
- Build an SEO command dashboard that tracks query clusters, landing pages, impressions, clicks, CTR, average position, conversions, indexing health, CWV, and publish/deploy annotations.
- Record every material SEO change so results can be tied back to launches.
- Define weekly or sprint-based experiments: title rewrites, internal-link changes, template changes, page consolidations, content refreshes, schema improvements, snippet engineering, or performance fixes.
- For each experiment, state the hypothesis, affected pages, expected metric movement, validation window, and decision rule.

PHASE 5 — RE-AUDIT AND ITERATE
- Re-crawl and re-audit after implementation.
- Compare pre-change vs post-change evidence.
- Find remaining weak pages, technical debt, or intent gaps.
- Keep iterating until there are no major unresolved blockers, no obvious thin/duplicate priority pages, no severe cannibalization, and no unowned high-intent clusters.
